Lattice-Driven Development

Why dependency ordering, verification gates, and topological execution beat hope-based AI workflows.

Joey Lopez

Why Lattice Verification Security
Section 01

Pipeline vs. Lattice

I spent years running pipelines. A→B→C. Fast feedback loop. Ship it. Find out what broke in production.

Then I hit a wall. One hallucination in step A silently corrupted everything downstream. By the time we caught it at C, the cascade had already spread. The problem: pipelines have zero verification. You only know if something is wrong after execution.

A lattice inverts this. Before C runs, it verifies that its input matches the contract defined by B. Before B runs, it verifies that its input matches A. Each layer is a proof, not a hope.

PIPELINE (hope-based): ┌───┐ ┌───┐ ┌───┐ │ A │ ────→ │ B │ ────→ │ C │ └───┘ └───┘ └───┘ ↓ Bug in production LATTICE (verified): ┌───┐ ✓ ┌───┐ ✓ ┌───┐ │ A │ ─→──→ │ B │ ─→──→ │ C │ └───┘ gate └───┘ gate └───┘ A→B? B→C? ↓ Bad state blocked
Warning

AI is fast but unreliable. It will confidently generate wrong output. If you run a pipeline, a hallucination becomes a bug in production. If you run a lattice, a hallucination becomes a broken edge that never executes. Fail at verification, not at scale.

That's the core distinction. Pipelines assume each step is correct. Lattices verify it. In a world where your agent is an LLM, lattices are the only way to stay sane.

Section 02

The Spec Folder

A lattice is physical. It lives in a folder. I call it spec/.

The spec folder is your ground truth. It contains five files, in dependency order:

spec/ ├── KNOWLEDGE.md ← Ground truth (human-written, verified once) ├── SPEC.md ← Contract (verified against KNOWLEDGE) ├── PLAN.md ← Execution order (verified against SPEC) ├── OUTPUT/ ← Generated artifacts └── EXECUTION.log ← Audit trail

Each file is a layer in the lattice. Each layer depends on the previous one. This creates a directed acyclic graph (DAG).

File Purpose Written By Verified By
KNOWLEDGE.md Immutable facts. What is true about the domain, prior art, constraints, invariants. Human Human code review
SPEC.md Contract. What the system will do. Acceptance criteria. Interface definitions. LLM + Human Human + KNOWLEDGE check
PLAN.md Execution roadmap. Decomposed tasks. Dependency graph. Build order. LLM + Human Human + SPEC check
OUTPUT/ Generated files. Code, configs, docs. One per task. LLM Human + PLAN check
EXECUTION.log Audit trail. Who did what, when, and why. System N/A (immutable)
Core Insight

KNOWLEDGE.md is the ground truth. Human-written, human-verified, never auto-generated. Everything downstream is checked against it. If KNOWLEDGE is solid, SPEC and PLAN can be verified mechanically. If KNOWLEDGE drifts, everything breaks.

Section 03

Verification Gates

Each layer has a verification gate. A gate is a test: "Does this layer comply with the contract defined by the previous layer?"

I don't do elaborate formal verification. I do manual spot-checks. But they're systematic:

  • KNOWLEDGE gate: Human reads it once. Is it factual? Is it complete? Sign off.
  • SPEC gate: Human + automated check. Does SPEC satisfy all constraints in KNOWLEDGE? No contradictions? Sign off.
  • PLAN gate: Human + automated check. Does PLAN cover all tasks in SPEC? Is the dependency graph acyclic? Can it execute top-to-bottom? Sign off.
  • OUTPUT gate: Human + automated check. Do the generated files match PLAN? Do they work? Can they be integrated? Sign off.
KNOWLEDGE.md ↓ (human review + sign-off) ✓ KNOWLEDGE verified ↓ SPEC.md ↓ (human + constraint check) ✓ SPEC verified against KNOWLEDGE ↓ PLAN.md ↓ (human + DAG check) ✓ PLAN verified against SPEC ↓ OUTPUT/ ↓ (human + integration test) ✓ OUTPUT verified against PLAN ↓ EXECUTION (immutable audit trail)
Try It

Take your current project. Can you draw the dependency graph? Can you list the verification gates? If not, you don't have a lattice—you have a pile. Start there.

The gates don't need to be fancy. A checklist in Markdown is enough. What matters is that each gate is explicit. You know what you're checking for, and you do it before proceeding.

Section 04

Topological Execution

Once the dependency graph is defined, the execution order is determined. This is topological sort.

Topological sort is how make works. It's how webpack works. It's how every build system works. You define the graph, the system figures out what to build first.

I apply the same principle to AI workflows. PLAN.md contains a dependency list. Each task lists its prerequisites. From this, I derive the topological sort—the correct build order.

PLAN.md example:

TASK 1: Write KNOWLEDGE.md
  Prerequisites: (none)

TASK 2: Write SPEC.md
  Prerequisites: TASK 1

TASK 3: Write PLAN.md
  Prerequisites: TASK 2

TASK 4: Generate code for Feature A
  Prerequisites: TASK 3

TASK 5: Generate code for Feature B
  Prerequisites: TASK 3

TASK 6: Integrate all outputs
  Prerequisites: TASK 4, TASK 5

Topological sort:
  1 → 2 → 3 → (4, 5 in parallel) → 6
Core Insight

The dependency graph determines the build order. You don't manually schedule tasks. You declare the graph, and the system executes it correctly. The same principle that makes build systems reliable makes LDD reliable.

This matters because it eliminates scheduling mistakes. If you forget that Task 4 depends on Task 3, you'll discover it in the verification gate, not in production.

Section 05

No Execution Path

Here's where LDD gets strange and powerful: the LLM never gets direct access to the shell.

In a typical workflow, you write a prompt, the LLM generates a bash script, and you run it. If the prompt is malicious or compromised, the LLM can execute arbitrary code. Guardrails try to prevent this by filtering bad patterns. But filtering is a game the attacker always wins.

LDD inverts this. The LLM generates files. A human reads them. The human decides whether to execute. The LLM's output is never executable—it's always a declaration.

Property Shell Script Workflow Lattice Workflow
No escalation path LLM → bash → system. Escalation is immediate. LLM → file → human review → execution. Human is the escalation gate.
Built-in audit trail History is implicit. What ran? No clear record. Every file, every gate, every execution is logged in EXECUTION.log. Full provenance.
Blast radius One bad script = system compromise. Radius = unbounded. One bad file = one human-reviewable decision. Radius = the scope of that one decision.
Execution inversion LLM decides what runs. Human trusts the LLM. Human decides what runs. LLM proposes, human disposes.
Warning

Regex-based guardrails can't stop certain attack classes. The answer isn't better filters. It's separating declaration from execution. If the LLM can't execute, it can't cause harm. Read guardrails-engineers.html for the full argument.

This is why I call it "no execution path." The LLM proposes, but it never executes. The human is always in the loop.

Section 06

Formal Foundations

LDD isn't just engineering intuition. It's grounded in formal methods and contract theory.

The spec is a contract. It defines the interface between the system and the user. It specifies preconditions (what must be true before execution), postconditions (what will be true after execution), and invariants (what is always true).

The lattice is a partial order. Dependency is a relation: A ≤ B means "A must complete before B." This induces a DAG. Topological sort finds a linear extension of this partial order—a valid execution sequence.

Convergence via constraints. Each layer (KNOWLEDGE → SPEC → PLAN → OUTPUT) narrows the solution space. KNOWLEDGE eliminates invalid domains. SPEC eliminates invalid contracts. PLAN eliminates invalid schedules. OUTPUT eliminates invalid implementations. By the time you execute, only correct solutions remain.

Core Insight

The spec isn't documentation. It's a constraint system. Each layer removes entropy from the solution space. Design by contract. Verify each layer. The result is a proof, not a hope.

I prefer not to formalize this further. The math is there if you need it (lattice theory, order theory, design by contract). But the practice is simple: define the graph, verify each layer, execute in topological order.

Section 07

Getting Started

You don't need to rewrite your entire workflow. Start with the next feature. Five concrete steps:

  1. Create spec/ folder. One folder per project.
  2. Write KNOWLEDGE.md. What is true about this domain? Constraints? Invariants? Prior art? One hour max. Human-only.
  3. Write SPEC.md. What will the system do? Acceptance criteria? Interface definitions? Ask an LLM to draft it, then verify against KNOWLEDGE. One hour max.
  4. Write PLAN.md. How will you build it? Decompose into tasks. Define dependencies. Ask an LLM to draft it, then verify against SPEC. Check that the graph is acyclic and executable. One hour max.
  5. Execute PLAN. For each task, ask the LLM to generate the output. Review it against PLAN. Run the gate check. Move on.
Try It

Pick your next feature. Before touching code, write the three files: KNOWLEDGE, SPEC, PLAN. Spend three hours. Then compare the result to how you usually build features. The difference is dramatic.

That's it. No fancy tooling. No formal verification. Just structure, gates, and honesty about what you know and what you don't.

The result: fewer bugs, faster development, and—most importantly—sleep. You know what your system will do before it does it.
