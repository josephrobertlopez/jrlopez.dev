Developer Second Brain

Interrogation-driven code assistance. Gather context first, generate structured plans and diffs — not raw code dumps.

context is everything structure gets rewarded you are the retrieval system

When to use each scenario

A — Migration

"Migrate from X to Y"

Framework upgrades, language bumps, major dependency swaps. Output: phased plan, diffs, rollback.

B — Refactoring

"Refactor this component"

Extract service, decompose monolith, improve testability. Output: dependency map, extraction plan.

C — Feature

"Implement following our patterns"

New feature matching existing conventions. Output: code stubs, test outline, integration points.

D — Debugging

"Track down this root cause"

Systematic production issue investigation. Output: hypothesis, diagnostic queries, solution options.

Interrogation questions

Phase 1 — Current State (5 questions)

  • Language + Framework + Version e.g. Python 3.11 + FastAPI
  • Architecture pattern Monolith, microservices, event-driven?
  • What specifically changes? One sentence.
  • Scope One file, one module, or multiple services?
  • Team size + your role IC, tech lead, etc.

Phase 2 — Target State (3 questions)

  • Success criteria Metrics, tests, deployment success?
  • Hard constraints No downtime, budget, timeline?
  • Dependencies What other systems does this touch?

Phase 3 — Team Context (4 questions)

  • Test coverage Existing patterns — unit, integration, e2e?
  • Conventions Naming, error handling, logging standards?
  • Review process Automated checks, approval gates?
  • Rollback plan Feature flag, parallel run, DB reversible?

The ReAct pattern — applied to code

THINK
Why are we making this change? What constraints exist? What could break?
ACT
The actual code change — as a diff, stub, or implementation with TODOs.
OBSERVE
How do we verify this works? Unit test, integration test, CLI command.

Example output — annotated code diff

# THINK: Why are we changing this?
# - Need async I/O for non-blocking behavior
# - Current version blocks request thread under load

# BEFORE
class DataFetcher:
    def get_data(self, url):
        return requests.get(url).json()

# AFTER
class DataFetcher:
    async def get_data(self, url: str) -> dict:
        async with aiohttp.ClientSession() as session:
            async with session.get(url) as response:
                return await response.json()

# OBSERVE: Tests to add:
# - Unit: mock async response, verify await
# - Integration: real async client, confirm non-blocking
# - Regression: callers still work (adapter if needed)

Pre-deployment checklist template

Verification Checklist
  • Tests pass (unit + integration + e2e)
  • Code review approved
  • No new linter / type-checker warnings
  • Dependency scan clean (no new CVEs)
  • Load test shows no degradation
  • Rollback plan documented
  • Migration is reversible (if DB changes)
  • Monitoring / alerting updated

Interrogation questions

Tree of Thoughts — ADR pattern

GENERATE
What are 3 fundamentally different approaches? Not variations — genuinely different directions.
EVALUATE
For each option: pros, cons, risks (High / Medium / Low). Be honest about what you're giving up.
DECIDE
Choose one. State the rationale: which constraint does this best satisfy? What risks are you accepting?

ADR template

Architecture Decision Record — ADR-NNN
## ADR-NNN: [Decision Title]

### Context
- Problem: [One sentence]
- Constraints: [Timeline, budget, team, compliance]
- Success criteria: [How will we know this was right?]

### Options Evaluated

Option A: [Approach Name]
Pros: aligned with team expertise, no migration needed
Cons: doesn't solve scale issue, adds coupling
Risks: [HIGH] can't grow beyond current hardware

Option B: [Approach Name]
Pros: industry-standard, phased migration possible
Cons: requires 2 new specialists, steep learning curve
Risks: [MEDIUM] budget overrun on hiring

### Decision
CHOOSE: Option [A/B]
Rationale: best fits [key constraint], team can execute in [timeline]

### Consequences
Positive: [specific wins]
Negative: [what we're giving up]
Action items:
- [ ] Spike: POC in 1 week
- [ ] Plan: Migration roadmap
- [ ] Team: Schedule training

Team standards — .cursorrules pattern

# .cursorrules — [Project] Team Standards

## Vision
"We are an async-first microservices team that values observable deployments."

## Core Patterns

# PATTERN: Structured logging (queryable)
logger.info("order_created", extra={
    "order_id": order.id,
    "user_id": order.user_id,
    "amount": order.total
})

# ANTI-PATTERN: String interpolation (not queryable)
logger.info(f"Order {order.id} created")  # Wrong

# PATTERN: Named exceptions with context
class PaymentProcessingError(Exception):
    def __init__(self, user_id: int, amount: float, reason: str): ...

# ANTI-PATTERN: Generic catch-all
except Exception as e:
    logger.error(f"Error: {e}")  # No context — useless in prod

# PATTERN: Versioned API contracts
@router.post("/v1/orders")
async def create_order(request: CreateOrderRequest) -> CreateOrderResponse: ...

## Code Review Checklist
# [ ] No sync I/O in async functions
# [ ] All exceptions named and contextual
# [ ] Logging is structured JSON
# [ ] Service boundaries have explicit contracts
# [ ] Tests cover happy path + 2 error scenarios
# [ ] Type hints on all signatures

Make Skills — Capstone

Stop using skills, start building them. Turn any repeated weekly task into a structured AI workflow you can save, share, and chain.

The capstone reveal: Every skill you build IS a RAG system. The interrogation questions = Retrieval. Your answers organized into sections = Augmentation. The AI generates from that structure = Generation. You've been building mini-RAG systems since the first exercise.

The 3-phase workflow

Phase 1 — Task Discovery (15 min)

  • Q1: Weekly task "What task do you repeat at least weekly?"
  • Q2: Current process "Walk me through your steps today, one by one."
  • Q3: Inputs "What information do you need before you start?"
  • Q4: Quality bar "What does GOOD output look like? What's BAD?"
  • Q5: Audience "Who uses your output, and how?"

Phase 2 — Pattern Extraction (10 min)

  • Q6: Category Code gen? Document creation? Analysis? Communication? Planning?
  • Q7: Context requirements "What 5-7 pieces of info always change but are always needed?"
  • Q8: Interrogation design "What questions should the skill ask to gather that context?"
  • Q9: Output structure "What sections, ordering, format does the output follow?"
Phase 3 is automatic. Feed your Phase 1-2 answers to any LLM and ask it to generate a SKILL.md in the format below. It will. The structure you gave it in Phase 2 is the spec.
